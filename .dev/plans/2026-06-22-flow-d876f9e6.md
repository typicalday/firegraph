# Fix: node:sqlite stripped from dist/sqlite/node-sqlite.{js,cjs}

## Goal

Restore the `node:` scheme on the `node:sqlite` import in the published build artifacts so consumers of `@typicalday/firegraph/sqlite-builtin` no longer hit `ERR_MODULE_NOT_FOUND` when they import the package. Add a regression guard that operates on the **built** output so this can never silently regress again.

## Root cause (confirmed read-only)

`tsup.config.ts` sets `target: 'node18'`. esbuild interprets this as a license to strip the `node:` scheme from built-in imports it recognises (e.g., `node:sqlite`) because that scheme was not universally supported before Node 14.18/16.0. `node:sqlite` is only available from Node 22.5, which means:

- esbuild has no entry for it in its built-in module table for the `node18` target.
- The import is **not** in the `external` array, so esbuild attempts to resolve it as a bare specifier `"sqlite"`.
- The emitted JS and CJS files contain `import from "sqlite"` / `require("sqlite")` — a module that does not exist anywhere in the dependency tree.

The existing unit tests (`tests/unit/sqlite-builtin*.test.ts`) import from **`../../src/sqlite/node-sqlite.js`** — the TypeScript source resolved through Vitest's transform pipeline — and never exercise the built artifacts, so the regression goes undetected by CI.

## Files to touch

### 1. `tsup.config.ts` — one-line change, the actual fix

Add `'node:sqlite'` to the `external` array. Place it adjacent to `'better-sqlite3'` and `'cloudflare:workers'` with a matching comment explaining it is a Node 22.5+ built-in.

Before:

```ts
external: [
  '@google-cloud/firestore',
  'json-schema-to-typescript',
  'react',
  'react-dom',
  'svelte',
  'ses',
  // Optional peer — only `firegraph/sqlite-local` imports it (dynamically).
  // Mark external so the native module is never bundled.
  'better-sqlite3',
  // Virtual workerd builtin — only resolvable inside Cloudflare Workers.
  'cloudflare:workers',
],
```

After (add immediately after the `better-sqlite3` block):

```ts
  // Node 22.5+ built-in — only resolvable at runtime, not at build time.
  // Without this entry esbuild (via the node18 target) strips the `node:` scheme
  // and emits bare `"sqlite"` which does not exist as an npm package.
  'node:sqlite',
```

No other change to `tsup.config.ts` is required. The `target` should remain `'node18'` — changing it would be a wider blast radius and is not needed.

### 2. New test file: `tests/unit/sqlite-builtin-dist.test.ts` — the regression guard

Create a new Vitest unit test that:

1. Runs `pnpm build` (or inspects the already-built artifacts) to obtain `dist/sqlite/node-sqlite.js` and `dist/sqlite/node-sqlite.cjs`.
2. Reads the text of each file and asserts that the string `'sqlite'` (bare, without the `node:` prefix) does NOT appear as a module specifier — specifically that neither file contains `from "sqlite"`, `require("sqlite")`, `from 'sqlite'`, or `require('sqlite')`.
3. Conversely asserts that `'node:sqlite'` DOES appear in each file (confirming the external was preserved).

**Implementation note:** The test must not invoke `pnpm build` itself (that would be slow and order-dependent in CI). Instead it should skip if the dist files are absent and assume the `build` CI job always runs before the artifact check. The simplest approach: the test file is placed under `tests/unit/` so it runs with `pnpm test:unit` but also guarded with `describe.skipIf(!existsSync(distPath))` so it silently skips in watch mode for developers who have not built yet. This mirrors the `HAS_NODE_SQLITE` skip-guard pattern already used in `tests/unit/sqlite-builtin.test.ts`.

Example structure:

```ts
import { existsSync, readFileSync } from 'node:fs';
import { join } from 'node:path';
import { describe, it, expect } from 'vitest';

const root = new URL('../../', import.meta.url).pathname;
const esmDist = join(root, 'dist/sqlite/node-sqlite.js');
const cjsDist = join(root, 'dist/sqlite/node-sqlite.cjs');
const hasDist = existsSync(esmDist) && existsSync(cjsDist);

describe.skipIf(!hasDist)('dist/sqlite/node-sqlite — node:sqlite import integrity', () => {
  for (const [label, path] of [
    ['ESM', esmDist],
    ['CJS', cjsDist],
  ]) {
    it(`${label}: does not contain bare "sqlite" module reference`, () => {
      const text = readFileSync(path, 'utf8');
      expect(text).not.toMatch(/from ["']sqlite["']/);
      expect(text).not.toMatch(/require\(["']sqlite["']\)/);
    });
    it(`${label}: preserves node:sqlite specifier`, () => {
      const text = readFileSync(path, 'utf8');
      expect(text).toMatch(/node:sqlite/);
    });
  }
});
```

### 3. `.github/workflows/ci.yml` — wire the regression guard into the build job

The existing `build` job runs `pnpm build` but does nothing with the output. Extend it to run the dist-integrity test immediately after the build. The test already skips gracefully when `dist/` is absent, so it is safe to add `pnpm test:unit tests/unit/sqlite-builtin-dist.test.ts` as an additional step after `pnpm build`.

No new CI job is required. The check runs inside the existing `build` job:

```yaml
- name: Build
  run: pnpm build

- name: Verify dist — node:sqlite import integrity
  run: pnpm test:unit tests/unit/sqlite-builtin-dist.test.ts
```

This ensures the regression guard runs against the actual built artifacts on every PR, while remaining in the same job so it shares the install cache.

## Verification steps (for the builder)

1. Run `pnpm build` in the worktree.
2. Inspect `dist/sqlite/node-sqlite.js` and `dist/sqlite/node-sqlite.cjs` — confirm `node:sqlite` is present and bare `sqlite` is absent.
3. Run `pnpm test:unit tests/unit/sqlite-builtin-dist.test.ts` — both ESM and CJS tests should pass.
4. Run `pnpm test:unit` — the existing `sqlite-builtin.test.ts` and `sqlite-builtin-search.test.ts` tests should remain green (they were not broken, only the dist output was).
5. Run `pnpm typecheck` — no change expected.
6. Run `pnpm lint` — no change expected (the new test file follows existing style; import sort from `eslint-plugin-simple-import-sort` applies).

## Assumptions

1. `target: 'node18'` is intentionally kept (downstream consumers may rely on it). The fix adds `node:sqlite` to `external` rather than raising the target to `node22`, which would be a wider breaking change.
2. The new dist-integrity test is placed in `tests/unit/` so it participates in `pnpm test:unit` without a new script. This matches the existing pattern for `sqlite-builtin.test.ts`.
3. The `hasDist` skip-guard is considered acceptable because the `build` CI job always runs before tests and the test is fast (file read + regex).
4. No change to the Vitest alias config is needed — the new test does not import from `cloudflare:workers`.
