### `src/sqlite/node-sqlite.ts`

**Module header / doc-comment**: Mirror the `local.ts` doc-comment adapted for node:sqlite. Note that `node:sqlite` is a Node built-in (>= 22.5) so no dynamic import is needed, and `DatabaseSync` is imported statically:

```typescript
import { DatabaseSync } from 'node:sqlite';
```

Note the Node >= 22.5 requirement and that this is the only module referencing `node:sqlite`.

**Key exports** (parallel to local.ts):

- `LocalNodeSqliteCapability` — type alias: `SqliteCapability | 'search.fullText' | 'search.vector'` (same as `LocalSqliteCapability` in local.ts)
- `NodeSqliteBackendOptions` — same interface as `LocalSqliteBackendOptions` (tableName, busyTimeoutMs, pragmas, fileMustExist, plus `SqliteBackendOptions` fields); accepts `DatabaseSync | string` for the db parameter
- `NodeSqliteBackend` — `{ backend: StorageBackend<LocalNodeSqliteCapability>; db: DatabaseSync; close(): void }`
- `createNodeSqliteExecutor(db: DatabaseSync): SqliteExecutor` — exported, parallel to `createBetterSqliteExecutor`
- `createNodeSqliteBackend(pathOrDb: string | DatabaseSync, options?: NodeSqliteBackendOptions): Promise<NodeSqliteBackend>` — the main factory

**Imports**: Same internal imports as `local.ts` — `FiregraphError`, `StorageBackend`, `createCapabilities`, `SqliteExecutor`, `SqliteTxExecutor`, `quoteIdent`, `validateTableName`, the full sqlite-search import set, `rowToRecord`, `FindNearestParams`, `FullTextSearchParams`, `StoredGraphRecord`, `SqliteBackendOptions`, `SqliteCapability`, `SqliteStorageBackend`, `createSqliteBackend`, `catalogTableName`. All with `.js` extensions. Also add `import { existsSync } from 'node:fs';` for the `fileMustExist` implementation.

**`createNodeSqliteExecutor(db: DatabaseSync): SqliteExecutor`**:

All methods are synchronous underneath wrapped in implicit Promise resolution:

```typescript
export function createNodeSqliteExecutor(db: DatabaseSync): SqliteExecutor {
  return {
    async all(sql: string, params: unknown[]): Promise<Record<string, unknown>[]> {
      return db.prepare(sql).all(...params) as Record<string, unknown>[];
    },
    async run(sql: string, params: unknown[]): Promise<void> {
      db.prepare(sql).run(...params);
    },
    async batch(statements): Promise<void> {
      // node:sqlite has no db.transaction() helper — use manual BEGIN/COMMIT/ROLLBACK.
      db.exec('BEGIN');
      try {
        for (const s of statements) {
          db.prepare(s.sql).run(...s.params);
        }
        db.exec('COMMIT');
      } catch (e) {
        db.exec('ROLLBACK');
        throw e;
      }
    },
    async transaction<T>(fn: (tx: SqliteTxExecutor) => Promise<T>): Promise<T> {
      db.exec('BEGIN IMMEDIATE');
      try {
        const result = await fn({
          async all(sql: string, params: unknown[]) {
            return db.prepare(sql).all(...params) as Record<string, unknown>[];
          },
          async run(sql: string, params: unknown[]) {
            db.prepare(sql).run(...params);
          },
        });
        db.exec('COMMIT');
        return result;
      } catch (err) {
        db.exec('ROLLBACK');
        throw err;
      }
    },
  };
}
```

Note: `batch` uses `BEGIN/COMMIT/ROLLBACK` (non-mechanical difference b from the proposal). `transaction` structure is identical to `local.ts`. `all` and `run` port 1:1.

**Private helpers**:

`isDatabase(value)` — detects a `DatabaseSync` by duck-typing:

```typescript
function isDatabase(value: unknown): value is DatabaseSync {
  return (
    typeof value === 'object' &&
    value !== null &&
    typeof (value as { prepare?: unknown }).prepare === 'function' &&
    typeof (value as { exec?: unknown }).exec === 'function'
  );
}
```

`PRAGMA_KEY_PATTERN` and `PRAGMA_VALUE_PATTERN` — copy verbatim from `local.ts`.

`applyPragmas(db: DatabaseSync, pragmas)` — identical validation logic but use `db.exec(\`PRAGMA ${key} = ${value}\`)`instead of`db.pragma(...)`. The key and value are validated against safe-identifier / safe-integer patterns before interpolation, so injection risk is zero. Same `FiregraphError` throws on bad input.

`registerVectorUdf(db: DatabaseSync)` — use `db.function(VECTOR_DISTANCE_UDF, { deterministic: true }, (stored, query, measure) => computeVectorDistance(stored, query, measure))`. This is identical to the better-sqlite3 API (non-mechanical difference c in proposal: UDF registration is 1:1). Wrap in try/catch to swallow re-registration.

`sweepOrphanedFtsArtifacts` — copy verbatim from `local.ts` (operates on `SqliteExecutor`, not the raw db).

`wrapLocalSearchBackend` — copy verbatim from `local.ts` (operates on `SqliteExecutor` + `SqliteStorageBackend`; return type becomes `StorageBackend<LocalNodeSqliteCapability>`).

**`createNodeSqliteBackend(pathOrDb, options = {})`**:

1. Destructure options: `{ tableName = 'firegraph', busyTimeoutMs = 5000, pragmas, fileMustExist, ...backendOptions } = options`.
2. If `typeof pathOrDb === 'string'`:
   - If `fileMustExist && !existsSync(pathOrDb)`: throw `new FiregraphError('createNodeSqliteBackend: file does not exist: ' + pathOrDb, 'INVALID_ARGUMENT')`. (The test does `rejects.toThrow()` with no code check, so any throw is fine; using `FiregraphError` is more consistent.)
   - Open: `db = new DatabaseSync(pathOrDb)`. Set `ownsDb = true`.
   - Apply WAL: `db.exec('PRAGMA journal_mode = WAL')` (non-mechanical difference a).
3. Else if `isDatabase(pathOrDb)`: `db = pathOrDb`, `ownsDb = false`.
4. Else: throw `FiregraphError('createNodeSqliteBackend expects a file path or an open DatabaseSync', 'INVALID_ARGUMENT')`.
5. `db.exec(\`PRAGMA busy_timeout = ${Math.max(0, Math.floor(busyTimeoutMs))}\`)` (always, same as local.ts).
6. If `pragmas`, call `applyPragmas(db, pragmas)`.
7. `registerVectorUdf(db)`.
8. Compose `optionsWithSearch` with `extraTableDDL` wrapping `buildLocalSearchDDL` exactly as in `local.ts`.
9. `const executor = createNodeSqliteExecutor(db)`.
10. `const inner = createSqliteBackend(executor, tableName, optionsWithSearch)`.
11. `const backend = wrapLocalSearchBackend(inner, executor, tableName)`.
12. `let closed = false;`
13. Return `{ backend, db, close() { if (closed || !ownsDb) return; closed = true; db.close(); } }`.

## TypeScript / @types/node Edge Case

The repo does NOT list `@types/node` in devDependencies (confirmed from `package.json`). TypeScript 5.7 does not include Node type definitions by default. `import { DatabaseSync } from 'node:sqlite'` will fail typecheck without `@types/node`.

**Builder action**: Before typechecking, run `pnpm add -D @types/node`. Check `tsconfig.json` for a `types` array — if it restricts to specific packages, add `"@types/node"` to that array too. `@types/node >= 22` includes `node:sqlite` definitions.

Check the tsconfig.json first to understand the current state. It may already be present in a way that isn't listed in package.json (e.g., a workspace-level dep), or the tsconfig may not need `types` augmentation.

## Edge Cases Summary

| Case                                        | Handling                                                                                           |
| ------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| `fileMustExist` on node:sqlite              | `existsSync()` check before `new DatabaseSync()` — throw `FiregraphError` if absent                |
| `:memory:` databases                        | `new DatabaseSync(':memory:')` — WAL pragma is no-op on memory DBs, same as better-sqlite3         |
| Caller-provided `DatabaseSync`              | `ownsDb = false`, WAL not applied, busy_timeout still applied                                      |
| Re-registering UDF on caller-provided db    | try/catch swallows duplicate-registration error                                                    |
| `db.open` property absent on `DatabaseSync` | Tests use query liveness instead of `.open` boolean                                                |
| Static import of `node:sqlite` in test      | Dynamic `import('node:sqlite')` inside test bodies, guarded by `describe.skipIf(!HAS_NODE_SQLITE)` |
| `@types/node` absent from devDeps           | Builder adds `pnpm add -D @types/node`                                                             |
| `node:sqlite` in tsup external list         | Not needed — Node built-ins are auto-external                                                      |
| `node:sqlite` available on CI Node 22       | GitHub Actions `ubuntu-latest` + `node-version: 22` ships Node 22.x which is >= 22.5               |
