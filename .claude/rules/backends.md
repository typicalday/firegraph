# Backends & Capabilities

The backend layer is split into per-edition modules. Each backend implements `StorageBackend` (`src/internal/backend.ts`) and exposes a phantom-typed `BackendCapabilities<C>` descriptor. The `C` parameter is a union of `Capability` literals that drives the conditional types in `GraphClient<C>` so methods only appear on the surface when the backend declares them.

The full design is `.claude/plans/backend-capabilities-plan.md`. This file is the day-to-day reference.

**`query.aggregate` sub-shape:** there is no separate sub-capability for the `min`/`max` subset. Both Firestore editions declare exactly `'query.aggregate'`; the runtime helper rejects `min`/`max` with `UNSUPPORTED_AGGREGATE`. SQLite and DO declare the same literal but accept the full set. Callers that depend on `min`/`max` should branch on the backend identifier, not on the capability flag.

**`search.fullText` and `search.geo` framing:**

- **Firestore Standard** does not support full-text search or geospatial queries at all. They are Enterprise-only product features. The "‚Äî" in this row will never become "‚úì".
- **Firestore Enterprise** supports both in production. As of `@google-cloud/firestore@8.5.0`, the typed Pipelines surface exposes `Pipeline.search({ query, sort?, addFields? })`, `documentMatches(query)`, `geoDistance(field, point)`, `score()`, and `BooleanExpression` / `Ordering` ‚Äî enough to wire FTS and geo without falling back to `rawStage(...)`. The Enterprise backend declares both capabilities and delegates to shared helpers `src/internal/firestore-fulltext.ts` and `src/internal/firestore-geo.ts`. Constraint: the `search()` stage **must be the first stage** of a pipeline (per the SDK docstring at `Pipeline.search`). Identifying filters (`aType` / `axbType` / `bType`) therefore go into a follow-up `where(...)` stage rather than the search query expression. The geo radius cap stays inside `search.query` (where the geo index applies it efficiently); ascending-distance ordering goes into `search.sort` when `orderByDistance !== false`. Migrations are not applied to the result ‚Äî the search index walked the raw stored shape, and rehydrating through the migration pipeline would change the candidate set the index already scored. Per-field FTS predicates (`matches(field, query)`) are not yet typed in 8.5.0; any non-empty `fields` list is rejected with `INVALID_QUERY` (`'fields is not yet supported'`) until the SDK exposes a typed per-field predicate.
- **SQLite / DO** never get these ‚Äî they have no native FTS or geo index, and emulating either over `json_extract` is not viable for any realistic dataset.

## Routing invariant

`createRoutingBackend` (`src/internal/routing-backend.ts`) enforces "declared capability ‚áí method exists" both directions:

- A backend that declares a capability MUST implement the corresponding method, or the routing wrapper throws at construction.
- A backend that implements an extension method but does NOT declare the matching capability is still callable through the routing wrapper, but the type-level surface won't expose it ‚Äî keep the two in sync.

`tests/unit/routing-backend.test.ts` pins both directions. If you add a new capability, update both `Capability` (in `src/types.ts`) and the matrix above.

## Editing rules

- **Adding a method to the type surface only.** Add the extension interface in `src/types.ts`. Do NOT declare the capability on any backend until a runtime path exists; declared-but-unimplemented capabilities turn the type gate into a runtime lie. See `FullTextSearchExtension` / `GeoExtension` for the pattern.
- **Adding a method to one backend.** Implement on the backend, declare the capability in its `BackendCapabilities` constant, and run `pnpm test:unit` ‚Äî the routing-backend invariant tests will catch any drift.
- **Sharing across Firestore editions.** Standard and Enterprise both delegate to helpers under `src/internal/firestore-*.ts`. Add the shared validation / path-normalisation logic there; backends should be thin wiring.
- **Sharing across SQLite editions.** Shared SQLite (`src/sqlite/`) and Cloudflare DO (`src/cloudflare/`) both compile through helpers under `src/internal/sqlite-*.ts`. Both editions dispatch through the single scope-free, table-name-parameterized compiler in `src/internal/sqlite-sql.ts` (`src/cloudflare/sql.ts` retains only the DO RPC wire helpers), and both go through `assertJsonSafePayload` (`src/internal/sqlite-payload-guard.ts`) for the JSON-safe payload guard at the write boundary.

## Edition-specific notes

- **firestore-enterprise:** Has its own `defaultQueryMode: 'pipeline' | 'classic'` toggle for the query path. Pipelines are the default; classic is forced when `FIRESTORE_EMULATOR_HOST` is set AND `FIRESTORE_EMULATOR_EDITION` is not `enterprise` (case-insensitive, whitespace-trimmed) ‚Äî firebase-tools v15.14+'s enterprise-edition emulator (`firebase emulators:start --database-edition enterprise`) DOES execute pipelines, so opting in via that env var keeps pipeline mode end-to-end against the local emulator. A one-time `console.warn` fires if classic is forced by the legacy/standard emulator OR if classic is forced in production. Transactions, writes, and doc lookups always use classic. The effective mode is exposed as `backend.queryMode` on the runtime backend type (`FirestoreEnterpriseBackend`, re-exported from `firegraph/firestore-enterprise`); keep a reference to the backend before passing it to `createGraphClient` since the public `GraphClient` surface does not expose `backend`. See `core-library.md` "Firestore Edition Internals."
- **firestore-enterprise ‚Äî `query.dml` preview gate:** `bulkDelete` / `bulkUpdate` dispatch through `Pipeline.delete()` and `Pipeline.update(AliasedExpression[])` stages, which are marked `@beta` in `@google-cloud/firestore@8.5.0`. The cap is gated on an explicit `previewDml: true` opt-in flag (default `false`); without it the cap stays off and the cascade falls back to the read-then-write `bulkRemoveEdges` path. A one-time `console.warn` fires on first construction with the flag. The flag is threaded through `subgraph()` so child backends inherit the cap. Pipeline DML stages run server-side regardless of `defaultQueryMode` ‚Äî even in classic-mode-forced environments the DML helpers still dispatch through Pipelines (the toggle only affects the read query path). The shared helpers (`src/internal/firestore-bulk-dml.ts`) reject empty filter lists with `INVALID_QUERY` and reject `deleteField()` sentinels in `bulkUpdate` patches (the typed `update(AliasedExpression[])` surface in 8.5.0 has no field-deletion transform); the `updatedAt` field is stamped client-side via `Timestamp.now()` since `update()` accepts no `FieldValue.serverTimestamp()` sentinel.
- **firestore-enterprise ‚Äî `query.join` wiring:** Pipeline mode collapses N sources into one round trip via `equalAny(field, sources)` (no 30-element cap). Classic mode (and Standard, always) chunks `'in'` over the documented 30-element cap with parallel `Promise.all` dispatch and post-concat re-sort to honour `orderBy`. NODE_RELATION self-loop guard (`aUid !== bUid`) is applied as a post-pass since classic queries can't express column-vs-column predicates and the typed Pipelines surface in 8.5.0 doesn't expose them either.
- **firestore-enterprise ‚Äî `traversal.serverSide` wiring (Phase 13c):** Multi-hop traversal compiles into a single nested Pipeline via `define(field('bUid').as('hop_0_join'))` + `addFields(child.toArrayExpression().as('hop_0_children'))`, with each hop's sub-pipeline's source predicate pinned via `equal(field('aUid'), variable('hop_{parentDepth}_join'))`. The whole tree dispatches in one round trip. All required primitives (`define`, `addFields`, `toArrayExpression`, `variable`) are GA-typed in `@google-cloud/firestore@8.5.0` (no `@beta` annotation), so unlike `query.dml` this cap is **not** behind a preview opt-in. Engine traversal always dispatches through Pipelines regardless of `defaultQueryMode` ‚Äî join-key binding via `define` / `variable` has no classic Query API equivalent. The compiler (`src/internal/firestore-traverse-compiler.ts`, pure / no SDK dependency) caps depth at `MAX_PIPELINE_DEPTH = 5` and rejects specs whose worst-case response size (`sources.length √ó ŒÝ(limitPerSource_i)`) exceeds the caller's `maxReads` budget; ineligible specs surface as `UNSUPPORTED_OPERATION` in `'force'` mode and silently fall back to the per-hop loop in `'auto'` mode (the default). The traversal layer (`src/traverse.ts`) refuses engine mode when any hop carries a JS `filter` predicate (server-side execution can't run JS) or when a hop has `targetGraph` set (cross-graph hops reach a different routed backend). The decoder strips `hop_{depth}_children` and `hop_{depth}_join` scaffolding before returning records and applies the NODE_RELATION self-loop guard for parity with `expand()`. Per-hop `truncated: true` is set when the returned edge count reaches `limitPerSource` (conservative: server hit its cap); `result.truncated` is `true` when any hop is truncated. Empty-source short-circuit fires at both client and executor levels. One server-side round trip is billed as `totalReads: 1` (mirrors the `expand()` fast-path accounting). Engine traversal is opt-in via `engineTraversal: 'auto' | 'force' | 'off'` on `TraversalOptions`; `'auto'` is the default.
- **firestore-standard:** Classic-only. No pipeline path, no emulator-fallback warning. Declares `query.join` via the same chunked `'in'` helper as Enterprise classic mode; does NOT declare `query.dml` (Pipeline DML stages are Enterprise-only by SDK design).
- **sqlite ‚Äî table-per-graph layout:** No `scope` column. The root graph lives in `tableName`; each subgraph gets its own physical table (`tableForScope` in `src/sqlite/catalog.ts`, injective name mangling) tracked in a `<root>_graphs` catalog. Schema (table + indexes + catalog) bootstraps lazily on first use; stale subgraph handles self-heal (re-bootstrap + retry once) when their table was dropped by a parent cascade. `findEdgesGlobal` is deliberately absent (no cross-table index) ‚Äî the client throws `UNSUPPORTED_OPERATION`. The better-sqlite3 file factory lives at `src/sqlite/local.ts`, published as `firegraph/sqlite-local` (`createLocalSqliteBackend`, `createBetterSqliteExecutor`); it is the only module that references `better-sqlite3` (optional peer dep, dynamic `import()`) ‚Äî never import it from `firegraph/sqlite` or any workerd-bundled path.
- **sqlite / cloudflare-do:** Both rely on SQLite 3.35+ for `DELETE ‚Ä¶ RETURNING` / `UPDATE ‚Ä¶ RETURNING` (used to return authoritative affected-row counts). All three target runtimes ‚Äî `better-sqlite3`, Cloudflare D1, DO `state.storage.sql` ‚Äî are well past that floor. See `cloudflare.md` for the workers-types and `cloudflare:workers` virtual-module rules.

## Exotic object keys: storable, but not filterable / indexable on SQLite / DO

A nested `data` key that isn't a plain code-style identifier ‚Äî anything not matching `/^[A-Za-z_][A-Za-z0-9_-]*$/`, i.e. a `generateId()`-shaped leading-digit key (`'4f9Kq_2bN'`), a key containing a literal dot (`'a.b'`), whitespace, brackets, quotes, etc. ‚Äî is handled asymmetrically across the write and read paths. Know the split before storing user-controlled or generated map keys.

**Writes accept exotic keys (all backends).** The deep-merge write contract (`updateNode` / `updateEdge`, merge-mode `putNode` / `putEdge`, Enterprise `bulkUpdate`) addresses each key as a literal path segment, never as parseable path syntax:

- **SQLite / DO** build the `json_set` / `json_remove` path via `buildJsonPath` (`src/internal/sqlite-data-ops.ts`) ‚Äî each segment is a `JSON.stringify`-quoted label (`$."4f9Kq_2bN"`, `$."a.b"`) bound as a SQL parameter, so dots stay inside the quotes and the literal key is written. `assertSafePath` (`src/internal/write-plan.ts`) only rejects empty-string segments.
- **Firestore** addresses each segment via a `FieldPath('data', ...segments)` object (per-document path) or the canonical backtick-escaped `FieldPath.formattedName` string (`bulkUpdate` alias, `buildDataPathAlias` in `src/internal/firestore-bulk-dml.ts`). Both treat the segment literally.

So an exotic-keyed record round-trips fine through write ‚Üí `getNode` / `getEdge` (which return the whole `data` blob).

**Reads / filters / projections / indexes reject exotic keys on SQLite / DO.** `compileFieldRef` (`src/sqlite/sql.ts`, `src/cloudflare/sql.ts`) splits the field on `.` and runs `validateJsonPathKey` per part, then **inlines** the path into the SQL text (`json_extract("data", '$.<path>')`). `sqlite-index-ddl.ts` (`CREATE INDEX ‚Ä¶ ON tbl(json_extract("data", '$.<path>'))`) uses the same inlined form and the same regex. The inlining is deliberate, not an oversight: SQLite's expression-index matcher requires the WHERE-clause path string to be byte-identical to the `CREATE INDEX` path string, and the quoted-label form used on the write path isn't used here. A non-matching segment throws `INVALID_QUERY` ("not a safe JSON-path identifier"). Consequence on SQLite / DO: an exotic key **cannot** appear as a `findEdges` / `findNodes` filter field, a `select` projection field, or an index field.

**Firestore has no such limitation** ‚Äî `findEdges` / `findNodes` filters address `data.*` fields via `FieldPath`, so exotic keys are filterable on both editions.

**Workaround when you need SQLite/DO filtering or indexing on a generated key:** prefix it into the safe range (e.g. `'h_' + generateId()`), or store the key as a _value_ under a fixed field and filter on that field instead of on the key.
